# Humanoid Clothing Tooling Research - 2026-05-27

## Goal

Move Reom/Anny/CharMorph humanoids from placeholder procedural boxes toward encounter-definition-driven fitted clothing that can be generated, cached, reused, and visually scored before entering the WebXR runtime.

## Recommended integration order

1. **Blender local garment fitter**
   - Build a local Python pipeline that imports a humanoid GLB, imports or creates a garment mesh, shrinkwraps the garment to the body, transfers weights from the body/armature, validates clipping, and exports GLB.
   - This is the fastest path because Blender is already in the repo workflow and can run headless.
   - Use for gowns, scrubs, jackets, masks, pediatric shirts, and role-specific accents.

2. **MakeHuman / MPFB clothing library**
   - Use permissive MakeHuman/MPFB clothing assets as the first real garment source library.
   - Retarget or fit assets onto Reom/Anny candidates through Blender shrinkwrap + data transfer.
   - Cache normalized garments by semantic key, for example `garment:clinical-scrub-top:adult:teal:v1`.

3. **CharMorph/Reom source body + fitted garment pass**
   - Keep Reom as the current best local body candidate, but stop baking planar garment boxes.
   - Add a post-generation clothing stage so each encounter actor gets garment meshes derived from actor role, environment, age, acuity, and scenario context.

4. **Cloth simulation refinement**
   - Add optional Blender cloth simulation for drape, then freeze/apply geometry before runtime export.
   - Use only when screenshot scoring flags boxiness or apron-like artifacts.

5. **Provider-gated advanced tools**
   - `marvelous_designer_manual_or_cli_requires_approval`: best-quality garment authoring/export candidate, but licensing/manual/paid workflow risk.
   - `clo3d_requires_approval`: similar high-quality garment ecosystem; keep gated.
   - `hunyuan3d_local`: useful for props/equipment and possible clothing accessories, but less reliable for rigged/fitted garments without Blender cleanup.
   - `meshy_cloud_requires_approval` / `tripo_cloud_requires_approval`: compare generated clothing or equipment only with explicit approval.

## Local pipeline design

1. Encounter definition produces actor wardrobe requirements.
   - Inputs: role, age band, body type, clinical setting, acuity, cultural modesty, scenario-required visual cues.
   - Example: OB patient -> `maternity triage gown`, `soft clinical robe`, `no generic scrub pads`, `allow abdomen silhouette`.

2. Asset factory resolves garment semantic key.
   - Check shared asset library first.
   - If cache hit, attach existing garment GLB/material preset.
   - If miss, enqueue garment work order.

3. Garment work order runs local Blender fitter.
   - Import humanoid source.
   - Import garment source mesh or generate a simple pattern mesh.
   - Fit with shrinkwrap/proportional scaling.
   - Transfer weights from nearest body mesh/armature.
   - Add material and metadata.
   - Export GLB with provenance.

4. Visual scoring gate captures WebXR-only screenshot.
   - Crop to WebXR pane.
   - Hide diagnostic overlays in source-clean mode.
   - Score clothing fit, clipping, body silhouette, role match, and pose.

5. Cache and reuse.
   - Store completed garment and actor composites in blob/local emulator asset library.
   - Track semantic keys, hashes, source provider, license, and LRU reuse.

6. Re-run Reom source comparator until the score reaches B+ or the next blocker is documented.

## Acceptance criteria for B+

- Clothing is a mesh fitted to the humanoid, not floating debug geometry or planar panels.
- No visible torso clipping in neutral front view.
- Arms/hands are relaxed enough to avoid a bind-pose read.
- Face, eyes, and mouth remain visible for dialogue quality review.
- Runtime source-clean screenshot contains only WebXR view and no UI side panel or debug scaffolding.
- Asset metadata identifies source, license, semantic wardrobe key, and cache status.
