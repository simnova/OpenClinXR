OpenClinXR

OpenClinXR is a Step 2 CS-inspired XR clinical skills exam platform for building, reviewing, and running timed clinical encounters from blueprint to runtime. It includes an encounter factory, clinical asset commons, capability arenas, and local/offline plus connected Azure deployment modes. OpenClaw is the build operating model behind the platform.

Platform overview

A production exam platform, encounter factory, asset commons, and capability arena.

Encounter Blueprint Factory

Reviewed encounter specifications (scenarios, actors, dialogue, emotion, traces, rubrics, asset needs) drive generated runtime, publication, launch selection, review packets, persistence, and evidence. Case definitions drive scenes; hand-built one-off rooms do not.

Clinical Asset Commons

Generated or approved assets such as rooms, floors, nurses, equipment, clothing, humanoids, animation clips, and provenance records can be reused across encounters instead of regenerated each time.

Capability Arena

Candidate technologies such as TTS, speech recognition, humanoid generation, IWSDK sidecars, and provider adapters compete in repeatable cage matches before production promotion.

Deployment modes

Local/offline first, connected when useful, Azure-ready for the production app surface.

Local/offline mode is the default development and validation path, using deterministic providers and preconfigured assets wherever possible. Connected mode can use approved provider/API adapters behind explicit gates. Azure is a target for API, orchestration, review, and deployment surfaces, while heavy local generation and capability cage matches stay separate until promoted.

How we build it

OpenClaw-style agentic execution keeps the product direction from drifting.

Repo-native OpenClaw is the build operating model, not an external daemon and not the product itself. It supports long-running, token-efficient autonomous work. Persistent indexed memory in agents/** + .agent-factory/. Snapshots + Efficiency Quick Refs in state files for low-token rehydration. Lease, drift/alignment guards, focused verification. Full iteration loop reserved for plateau/synthesis. Live subagents or local role consultation. Always record in canonical state only.
