# OpenClaw Drift Police Memory

## Durable Lessons

- 2026-05-27 hardening established that OpenClaw drift should be caught by `pnpm docs:drift-check`, `pnpm agent:alignment`, the doc authority registry, the generated artifact registry, and the protected runbook before a long unattended batch begins.
- The main recurring failure mode is not lack of work; it is doing safe, concrete work that does not advance the blueprint-driven encounter factory, such as stale evidence refreshes, one-off screenshot loops, or hand-authored scene/clothing fixes.
- Corrections should be small and repo-native: register or remove artifacts, move useful context into canonical state, then pivot to a product-building slice.

## Active Risks

- Agents may treat the Drift Police as a product implementer. It is not; it should identify drift and route work back to the coordinator and worker.
- Agents may pass unit tests while still drifting away from end-to-end running software evidence. The correction is to require focused runtime/browser/API evidence when the slice touches runtime, UI, XR, scene, asset, or factory behavior.

## Preferred Review Heuristics

- Ask: does this slice strengthen the case-definition-to-runtime factory, conversation tooling, reusable asset pipeline, review/persistence path, or focused verification of touched behavior?
- Ask: would this artifact still make sense to a future agent after context compaction?
- Ask: is this a canonical state update, a registered evidence artifact, or clutter?
- Ask: is a one-off encounter improvement being converted back into reusable factory/provider/cache metadata?
